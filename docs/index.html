Quick start

Run the latest Qiskit (…) in your browser via Binder — no install:

Launch latest-small in Binder Launch latest-xl in Binder

Or pull with Docker: 

docker run --rm -p 8888:8888 ghcr.io/janlahmann/qiskit:latest-small

small = lean (qiskit + aer + ibm-runtime). xl = broad, with addons + scientific stack + nbgitpuller + single-file ?fromURL= launcher. Full details in the README.

Open a notebook in Binder

Two ways to launch — pick the one that matches what you need to open. Both produce a Binder URL you can copy and share.

Clone whole GitHub repo Open one notebook by URL
What gets fetched The entire repository (every file at HEAD), via nbgitpuller. Just the single .ipynb file, via jupyterlab-open-url-parameter.
Use when The notebook needs sibling files (data, images, helper modules) — or you want a working copy with git pull updates. The notebook is self-contained (only standard imports, no relative open(…) calls).
Cold-start cost Image pull + repo clone. Large repos (e.g. Qiskit/documentation, hundreds of MB) take 30–60s on top of the image. Image pull + a single ~50 KB file fetch (~10s after image is hot).
Image flavors Any image (small or xl). xl only — small doesn't ship the jupyterlab-open-url-parameter extension.

All images

Every image is daily-rebuilt, Trivy-scanned (HIGH/CRITICAL gated), and cosign-signed via GitHub Actions OIDC.

Qiskit Flavor Binder Docker

    How it works

    Dockerfile is parameterised by Qiskit version × flavor and installs the matching versions/<target>/requirements.txt. A GitHub Actions workflow builds per-arch (amd64 + arm64), gates on a Trivy scan, publishes the multi-arch manifest to GHCR with a SLSA provenance attestation, signs it with cosign, and force-pushes a tiny stub branch that mybinder.org consumes to skip the dependency-tree rebuild.

    A daily cron rebuilds everything against the current base image to absorb upstream CVE fixes. Dependabot watches the docker base, GitHub Action versions, and the latest Qiskit minor's pip pins.
